## 한 줄 정의

**신제품 기획 단계에서 부품 사양과 작동 환경을 함께 사전 시뮬레이션하여, 개발 착수 전 최적 사양을 수렴하는 사내 도구.**

결과물 = 개발팀에 전달할 확정 사양서 (Spec Sheet).

---

## 무엇을 하는가

### 입력 (사용자가 정의)
- **제품 부품 사양** — 베이스 폼/높이/무게, 팔 개수·길이·DOF, 액추에이터 모델, 엔드이펙터
- **작동 환경** — 방 평면도, 가구 배치(소파/식탁/싱크대/책상/의자), 바닥 장애물(러그/문턱/케이블)
- **타겟 작업** — 환경 내 특정 위치의 물체 (예: 식탁 위 컵, 소파 위 리모컨)

### 출력 (도구가 자동 산출)
- **공학적 분석** — 도달 영역, 페이로드 한계, 관절 토크, ZMP 안정성
- **환경 적합성** — 각 타겟에 대한 도달 가능성 + 사유, 베이스 통과 가능 영역
- **사양 후보 비교** — 후보안 A/B/C 정량 비교 매트릭스
- **공학적 검토 의견** — Claude API 기반 자연어 코칭
- **사양서 PDF** — 개발팀 전달용 확정 Spec Sheet

---

## 무엇이 다른가

| 기존 방식 | ARGOS-Designer |
|---|---|
| 컨셉 → 회의 → CAD 모델링 → 계산 → 시제품, 4~6주 | 컨셉 → 도구 입력 → 자동 분석 → 후보 비교 → 사양서, **3~5일** |
| 사양 변경 1건 = 3~5일 | 사양 변경 1건 = **30분~1시간** |
| 다중 컨셉 후보 비교는 시간 부족으로 거의 안 함 | **5~10개 후보 동시 비교 가능** |
| 환경 시뮬은 별도 도구 (Webots/Gazebo/MuJoCo)에서 연구원이 별도 셋업 | **부품과 환경을 한 도구에서 결합 분석** |

---

## 진짜 차별점 — 부품 × 환경 결합 분석

기존 OSS 도구(URDF-Studio, Robosynx, MuJoCo)는 *부품 정의*에 강하고, 환경 시뮬 도구(Webots, Gazebo)는 *환경*에 강합니다. **둘을 한 화면에서 결합해서 "이 사양으로 이 환경에서 이 작업이 가능한가"를 즉시 답하는 도구는 시장에 없습니다.**

이게 LG의 위치 — 청소기·가전 도메인 + 휴머노이드 IP를 동시에 가진 — 에서 만들 가치가 있는 정확한 자리.

---

## 첫 유즈케이스 (Phase 1)

**로봇청소기 + 매니퓰레이터 1~2개**

이유:
- LG 도메인과 직결 (로보킹/코드제로 + CLOiD)
- 차별점이 명확 (작은 베이스에 팔 달면 ZMP·전복 안정성이 즉시 핵심 변수)
- 사용자(기획자)가 답을 듣고 싶은 질문이 구체적 ("소파 위 컵을 잡을 수 있나, 그러려면 사양이 어때야 하나")

휴머노이드 전체로 확장은 Phase 3 이후.

- 사양 후보 3개 이상 비교 실시 누적 30건 이상
